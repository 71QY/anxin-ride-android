## 重点 1：AI 意图识别的准确性与多轮对话上下文

**难点：**
- 老年人语言表达口语化严重（如"我要去那个大医院"），传统关键词匹配准确率低。
- 需理解多轮对话中的指代关系（如"第一个"、"换一个"），并防止 AI 产生"自动下单"等危险幻觉。

**解决方案：**
1. **通义千问语义理解引擎**：接入阿里云 DashScope (qwen-turbo)，通过精心设计的 System Prompt 约束输出 JSON 格式，强制区分 `SEARCH`、`ORDER`、`CHAT` 意图。
2. **历史上下文注入**：利用 Redis 存储最近 10 条会话消息，在请求 AI 时动态注入，实现对"就这个"、"远不远"等指代词的精准理解。
3. **方言翻译中间层**：实现 `DialectTranslationService`，在意图识别前将粤语、四川话等方言文本实时翻译为普通话，提升模型识别率。
4. **安全防误触机制**：在 Prompt 中设置硬性约束，`autoOrder` 字段默认强制为 `false`，确保所有订单必须经过用户二次确认，杜绝意外叫车。

## 重点 2：多源地图数据融合与降级策略

**难点：**
- 单一地图源（如仅用高德）在偏远地区或特定 POI 搜索上存在盲区。
- 腾讯与高德 API 返回格式不统一，且坐标系存在差异，直接混合会导致定位漂移。

**解决方案：**
1. **腾讯优先 + 高德兜底策略**：设计 `SearchStrategyEngine`，优先调用腾讯地图 WebService API。若结果不足 3 个，自动触发高德地图补充搜索，最大化覆盖率。
2. **标准化 POI 数据模型**：建立统一的 `PoiDTO` 结构，在后端完成坐标系统一转换（WGS84）和字段清洗，前端无需处理复杂的坐标偏移。
3. **智能去重与纠偏**：使用 `name + lat + lng` 组合作为唯一标识进行去重；建立口语纠偏字典（如"老校区"映射为"韩山师范学院东区"），提升搜索命中率。

### ✅ 前端实现补充

**前端职责：**
- **高德地图 SDK 集成**：使用 `amap-full-11.1.0.aar` 完整包,配置 Android Key 和 Web Key
- **多场景定位策略**：前台高精度 GPS + 网络定位,后台 `LocationTrackingService` 持续追踪
- **地图相机优化**：初始化时使用 `moveCamera` 而非 `animateCamera`,默认 zoom=16.5f(街道级)
- **POI 列表渲染**：前端只负责渲染后端返回的 POI 列表,不参与排序算法
- **点击地图选点**：支持长按地图获取当前位置坐标

## 重点 3：POI 智能排序算法与同城加权

**难点：**
- 简单的距离排序无法满足老人需求（如搜索"大学"，附近的"理发店"不应排在前面）。
- 需平衡名称相关性、类型匹配度与物理距离之间的关系。

**解决方案：**
1. **多维度分层评分算法**：采用 `名称匹配(50%) + 类型匹配(30%) + 距离分数(20%)` 的加权公式。
2. **匹配等级机制**：针对名称匹配细化为完全匹配（100分）、开头包含（80分）、地址包含（30分）四个等级。
3. **知名地标特殊权重**：对故宫、北大等 50+ 知名地标实施特殊权重策略，名称权重提升至 80%，确保地标优先展示。
4. **同城加成系数**：通过逆地理编码判断 POI 与用户是否属于同一城市，同城地点得分乘以 1.2 倍系数，过滤异地干扰项。

### ✅ 前端实现补充

**前端职责：**
- **POI 列表展示**：接收后端返回的已排序 POI 列表,渲染为可点击卡片
- **用户选择交互**：用户点击 POI 卡片后,发送确认消息给后端
- **地图标记渲染**：在地图上显示 POI 位置标记,支持点击查看详情

## 重点 4：图片识别与地址类智能判断

**难点：**
- 如何准确判断一张图片是否为"地址类"（如路牌、招牌），以及如何处理 OCR 识别后的杂乱文字。

**解决方案：**
1. **多模态模型应用**：使用通义千问 qwen-vl-plus 模型进行 OCR 识别与语义分析，支持 Base64 图片流直接输入。
2. **地址类特征提取**：制定双重判断规则——OCR 文字包含地址关键词（路、街、号、市等）或 AI 意图直接判定为 `SEARCH` 类型。
3. **图片匹配打分**：将 OCR 提取的文字与地图 POI 进行模糊匹配，根据名称重合度给予 600-1000 分的差异化权重，实现"拍图即搜"。

### ✅ 前端实现补充

**前端职责：**
- **图片压缩策略**：OCR 场景最大 1920x1080 质量 85%,聊天图片动态压缩到 800KB 以内
- **Base64 编码优化**：移除 `data:image/jpeg;base64,` 前缀,异步执行编码避免阻塞 UI
- **多图批量发送**：待发送列表最多 3 张,底部预览栏显示缩略图支持删除
- **OCR 结果展示**：后端返回 `{ocrText, places, needConfirm}`,前端提取文本并渲染 POI 卡片
- **内存管理**：图片解码后立即释放原始 Bitmap,使用 `remember` 缓存解码结果

## 重点 5：WebSocket 实时通信与亲情守护同步

**难点：**
- 代叫车场景下，长辈端、亲友端、司机端（模拟）三方状态需毫秒级同步。
- 长连接在弱网环境下的稳定性与断线重连。

**解决方案：**
1. **双重推送机制**：后端在订单状态变更（接单、位置更新、到达）时，检查 `proxyUserId`，同时向长辈和亲友的 Session 推送消息，实现行程实时共享。
2. **指数退避重连**：前端实现 1s/2s/4s/8s 指数退避重连策略，配合 OkHttp 内置 Ping 机制，确保连接高可用。
3. **标准化消息协议**：定义统一的 `AgentMessage` 结构，支持 `search/order/guard_push/image` 等多种业务类型，前端通过 `type` 字段路由到不同 UI 组件。

### ✅ 前端实现补充

**前端职责：**
- **单例 WebSocket 客户端**：`@Singleton` 注解确保全局唯一实例,HomeViewModel 和 ChatViewModel 共享同一连接
- **智能重连机制**：指数退避策略(1s→2s→4s→8s→16s),最多重试 5 次,连接中状态标记防止重复连接
- **心跳保活**：OkHttp 内置 `pingInterval(30, TimeUnit.SECONDS)`,前端额外发送 ping 消息
- **消息解析容错**：使用 `kotlinx.serialization` 的 `ignoreUnknownKeys = true`,FastJSON 引用格式兼容
- **亲情守护推送处理**：监听 `GUARD_PUSH` 消息类型,弹出代叫车确认对话框

## 重点 7：高并发场景下的 API 限流与性能优化

**难点：**
- 多个 POI 并行计算路线耗时过长，容易触发地图 API 的 QPS 限制。

**解决方案：**
1. **并行路线计算**：使用 `CompletableFuture` 配合自定义线程池（核心 10 线程），对搜索结果中的多个 POI 同时发起路线规划请求，将耗时从 10 秒压缩至 2-3 秒。
2. **智能缓存策略**：利用 Redis 缓存 POI 搜索结果（10分钟）和会话状态（30分钟），减少重复 API 调用。
3. **超时降级控制**：并行任务设置 10 秒超时阈值，超时后自动返回已有结果，防止因个别 API 响应慢导致整体服务卡顿。

### ✅ 前端实现补充

**前端职责：**
- **Compose 性能优化**：
  - 使用 `remember` 缓存计算结果,避免重复计算
  - `LazyColumn` 替代传统 ListView,支持大量消息高效滚动
  - `derivedStateOf` 派生状态,减少不必要的重组
  - `collectAsStateWithLifecycle` 生命周期感知,避免内存泄漏
  
- **图片性能优化**：
  - 异步压缩和编码,避免阻塞 UI 线程
  - 使用 `remember` 缓存解码结果,避免重复解码
  - 大图查看器支持滑动关闭,及时回收资源
  
- **网络性能优化**：
  - OkHttp 连接池复用,减少握手开销
  - WebSocket 长连接,避免频繁建立 HTTP 连接
  - 请求去重,避免重复调用相同接口
  
- **TTS 语音播报**(可选功能):
  - 使用 Android 系统 `TextToSpeech` API
  - 仅在长辈模式下启用语音播报
  - 初始化时设置中文语言支持
  - 异常捕获,失败不影响其他功能
