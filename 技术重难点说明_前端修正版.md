## 重点 1:双引擎语音识别与方言支持

### ✅ 实际实现(前端)

**难点:**
- 用户输入口语化严重,需要准确识别目的地意图
- 不同地区用户需要使用方言进行语音输入
- 实时显示识别结果,提升用户体验

**解决方案:**
1. **双语音识别引擎架构**
   - **百度语音 SDK**(主用):国内支持好、免费额度大、无需 Google 服务
   - **讯飞语音 SDK**(备用):作为降级方案,提高可用性
   - 两个引擎独立初始化,互不影响

2. **方言支持机制**
   - 支持普通话、粤语、潮汕话、客家话等多种方言
   - 通过 `language` 和 `accent` 参数传递给语音 SDK
   - UI 提供方言选择对话框,用户可自由切换

3. **实时部分结果展示**
   - 监听 `onPartialResult` 回调,逐字显示正在说的话
   - 类似微信语音输入的交互体验
   - 最终结果通过 `onResult` 回调获取完整文本

4. **JSON 解析容错**
   - 百度语音返回格式:`{"ws":[{"cw":[{"w":"文字"}]}]}`
   - 讯飞语音返回格式:类似但字段名可能不同
   - 统一的 `parseVoiceResult()` 方法处理两种格式
   - 添加空值检查和异常捕获,避免崩溃

## 重点 2:高德地图 SDK 集成与定位服务

### ✅ 实际实现(前端)

**难点:**
- 高德地图 SDK 版本兼容性(API Level 26+)
- 实时定位精度与电量消耗平衡
- 地图相机平滑移动与缩放控制
- 后台定位追踪服务保活

**解决方案:**
1. **高德地图 SDK 11.1.0 集成**
   - 使用 `amap-full-11.1.0.aar` 完整包
   - 配置 Android Key 和 Web Key(逆地理编码)
   - SHA1 签名指纹绑定(调试版 + 发布版)

2. **多场景定位策略**
   - **前台定位**:高精度 GPS + 网络定位,实时更新
   - **后台定位**:`LocationTrackingService` 前台服务,持续追踪
   - **单次定位**:点击地图时获取当前位置
   - 权限分级处理:始终允许/仅使用时允许

3. **地图相机优化**
   - 初始化时使用 `moveCamera` 而非 `animateCamera`,提升速度
   - 默认 zoom=16.5f(街道级),避免显示空茫茫的城市级别
   - 支持手势缩放、双击放大、长按选点

4. **POI 搜索与展示**
   - 调用后端 WebSocket 接口获取 POI 列表
   - 前端只负责渲染,不参与排序算法
   - 支持点击地图选点和关键词搜索两种方式

## 重点 3:WebSocket 实时通信与会话管理

### ✅ 实际实现(前端)

**难点:**
- 长连接状态管理与自动重连
- 多种消息类型的统一解析
- 心跳机制保持连接活跃
- 跨 ViewModel 共享 WebSocket 连接

**解决方案:**
1. **单例 WebSocket 客户端**
   - `@Singleton` 注解确保全局唯一实例
   - 通过 Hilt 依赖注入到各个 ViewModel
   - HomeViewModel 和 ChatViewModel 共享同一连接

2. **标准化消息协议**
   - 请求格式:`{type, sessionId, content/imageBase64, lat, lng, timestamp}`
   - 响应格式:`{success, type, message, data, places, needConfirm, code}`
   - 支持类型:`chat/search/order/error/image_result/guard_push`

3. **智能重连机制**
   - 指数退避策略:1s → 2s → 4s → 8s → 16s
   - 最多重试 5 次,失败后提示用户
   - 连接中状态标记(`isConnecting`),防止重复连接
   - 正常关闭不重连,异常关闭才重连

4. **心跳保活**
   - OkHttp 内置 `pingInterval(30, TimeUnit.SECONDS)`
   - 前端额外发送 `{type:"ping", sessionId, timestamp}` 消息
   - 30 秒内只打印一次 connected 日志,避免刷屏

5. **消息解析容错**
   - 使用 `kotlinx.serialization` 的 `ignoreUnknownKeys = true`
   - FastJSON 引用格式兼容(`$ref` 循环引用)
   - 未知消息类型不崩溃,记录日志即可

## 重点 4:图片识别与多模态交互

### ✅ 实际实现(前端)

**难点:**
- 图片压缩与 Base64 编码性能优化
- 多张图片批量上传(最多 3 张)
- 大图预览与内存管理
- OCR 结果展示与 POI 匹配

**解决方案:**
1. **图片压缩策略**
   - OCR 场景:最大 1920x1080,质量 85%
   - 头像上传:最大 500x500,质量 85%
   - 聊天图片:动态压缩到 800KB 以内
   - 使用 `Bitmap.createScaledBitmap` 保持宽高比

2. **Base64 编码优化**
   - 移除 `data:image/jpeg;base64,` 前缀后再编码
   - 异步执行压缩和编码,避免阻塞 UI 线程
   - 计算压缩前后大小,日志输出便于调试

3. **多图批量发送**
   - 待发送列表(`pendingImages`),最多 3 张
   - 底部预览栏显示缩略图,支持删除
   - 一次性发送所有图片,附带文字说明
   - 发送后清空列表,避免重复

4. **OCR 结果处理**
   - 后端返回 `{ocrText, places, needConfirm}`
   - 前端提取 OCR 文本并显示
   - 如果有 POI 列表,渲染为可点击卡片
   - 非地址类图片返回自然语言描述

5. **内存管理**
   - 图片解码后立即释放原始 Bitmap
   - 使用 `remember` 缓存解码结果,避免重复解码
   - 大图查看器支持滑动关闭,及时回收资源

## 重点 5:长辈模式与适老化设计

### ✅ 实际实现(前端)

**难点:**
- 长辈用户操作能力弱,需要简化流程
- 大字体、高对比度、简化操作流程
- 代叫车服务与远程协助
- 长辈端与普通端界面隔离

**解决方案:**
1. **双界面架构**
   - `ElderSimplifiedScreen.kt`:长辈专属界面
   - `NormalHomeScreen.kt`:普通用户界面
   - 通过 `isElderMode` 标志位切换
   - 修改一个文件不会影响另一个

2. **适老化 UI 设计**
   - 标题字号 30sp(普通端 18sp)
   - 按钮高度 72dp(普通端 48dp)
   - 圆角半径 20dp,视觉更柔和
   - 高对比度配色(蓝色 #1677FF)
   - 居中标题,减少认知负担

3. **快捷操作区域**
   - "呼叫亲友"按钮:一键拨打绑定的守护者电话
   - "智能体对话"按钮:进入语音助手聊天
   - 全屏模式切换:隐藏顶部栏,专注地图
   - 底部面板紧凑布局,减少滚动

4. **代叫车确认机制**
   - 亲友发起代叫车请求
   - WebSocket 推送 `GUARD_PUSH` 消息
   - 长辈端弹出确认对话框(大按钮)
   - 两次确认防止误操作

5. **亲情守护管理**
   - 普通用户:可查看我的长辈列表
   - 支持解绑、添加新长辈
   - 长辈信息:姓名、手机号、关系
   - 独立页面:`GuardianManagementScreen.kt`

## 重点 6:Jetpack Compose 现代化 UI 架构

### ✅ 实际实现(前端)

**难点:**
- 声明式 UI 的状态管理
- 复杂列表性能优化
- 键盘弹出时布局调整
- 深色模式适配

**解决方案:**
1. **MVVM + Clean Architecture**
   - View:Jetpack Compose UI
   - ViewModel:`StateFlow` + `collectAsStateWithLifecycle`
   - Model:Data class + Repository
   - 单向数据流,状态不可变

2. **Hilt 依赖注入**
   - `@HiltAndroidApp` 初始化
   - `@AndroidEntryPoint` 注入 Activity/Fragment
   - `@Inject constructor` 注入 ViewModel
   - `@Module` + `@Provides` 提供单例

3. **性能优化**
   - `LazyColumn` 替代传统 ListView
   - `remember` 缓存计算结果
   - `derivedStateOf` 派生状态
   - `LaunchedEffect` 副作用管理

4. **导航与路由**
   - Navigation Compose 管理页面跳转
   - 深链接支持:`navigate_to_chat` 参数
   - 返回栈管理,避免内存泄漏

5. **主题与样式**
   - Material3 设计系统
   - 动态颜色(Dynamic Color)
   - 深色模式自动适配
   - 自定义 Typography 和 Shape

## 重点 7:Hilt 依赖注入与模块化设计

### ✅ 实际实现(前端)

**难点:**
- 单例生命周期管理
- 跨模块依赖注入
- kapt 编译性能优化
- VerifyError 问题解决

**解决方案:**
1. **依赖注入架构**
   - `SingletonComponent`:应用级单例
   - `ViewModelComponent`:ViewModel 作用域
   - `ActivityRetainedComponent`:Activity 保留作用域

2. **网络模块示例**
   ```kotlin
   @Module
   @InstallIn(SingletonComponent::class)
   object NetworkModule {
       @Provides
       @Singleton
       fun provideApiService(): ApiService {
           return RetrofitClient.instance
       }
       
       @Provides
       @Singleton
       fun provideChatWebSocketClient(): ChatWebSocketClient {
           return ChatWebSocketClient()
       }
   }
   ```

3. **kapt 优化配置**
   - `correctErrorTypes = true`
   - `useBuildCache = true`
   - `-Adagger.fastInit=enabled`
   - `-Xmaxerrs 500`

4. **VerifyError 修复**
   - 禁用增量编译:`kotlin.incremental=false`
   - 跳过预发布检查:`-Xskip-prerelease-check`
   - 允许不稳定依赖:`-Xallow-unstable-dependencies`
   - 清理 `.gradle` 缓存后重新编译
